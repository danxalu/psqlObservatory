Уровень 1

Мониторинг операционной системы и хоста

Это самый внешний слой. Здесь мы выясняем, хватает ли серверу CPU, памяти, IOPS, пропускной способности диска и сети. PostgreSQL часто выглядит «медленным» только потому, что ОС уже задыхается от ожидания диска, swap или сетевой деградации.

top htop vmstat iostat pidstat ss

Теоретический блок

Категория инструментов: системные утилиты Linux/Unix и сборщики статистики, работающие вне PostgreSQL.
Ключевая идея: не пытаться объяснить любую деградацию через SQL, пока не проверены ресурсы узла.
Конкретные средства: top/htop для процессов, vmstat для памяти и scheduler, iostat -xz для диска, pidstat для профиля postgres-процессов, ss и sar -n для сети.
Что важно интерпретировать: iowait показывает долю времени CPU в ожидании I/O, а await в iostat помогает понять задержку дисковых операций. Если оба показателя нормальные, диск не выглядит главным узким местом.

Визуальный блок

top - 16:33:17 up 14 days, load average: 5.84, 5.12, 4.88 %Cpu(s): 28.1 us, 10.2 sy, 33.4 id, 27.2 wa MiB Mem : 15894 total, 722 free, 11138 used, 4034 buff/cache MiB Swap: 2048 total, 1430 used PID USER S %CPU %MEM COMMAND 2134 postgres S 89.2 13.1 postgres 2135 postgres S 74.0 12.7 postgres
Здесь видно типичный паттерн: рост swap и disk await совпадает с жалобами на медленные запросы. Это сигнал сначала проверить ресурсную базу, а не менять индексы вслепую.

Практический блок

top -Hp $(pgrep -d, postgres) vmstat 1 free -h Интерпретация: • высокий us → CPU уходит в вычисления запросов • высокий wa → процессоры простаивают, ожидая диск • swap under load → памяти не хватает, latency растет
iostat -xz 1 pidstat -d 1 Проверяем: • await • %util • read/write throughput • всплески во время checkpoint или bulk load
ss -tpn | grep 5432 sar -n DEV 1 Ищем: • backlog по TCP • retransmits • saturation канала • долго висящие client connections
Полезная привычка: записывайте время всплеска и сразу сверяйте его потом с метриками PostgreSQL. Так легче строить причинно-следственную цепочку между инфраструктурой и SQL-нагрузкой.

Уровень 2

Внешние системы мониторинга: Prometheus, Grafana, Alerting

На этом уровне метрики превращаются в временные ряды, графики и алерты. Здесь удобно видеть историю, а не только текущее состояние, и сопоставлять инфраструктуру с состоянием PostgreSQL в один и тот же момент.

Prometheus Grafana postgres_exporter Alertmanager Zabbix

Теоретический блок

Prometheus забирает метрики по HTTP с exporters, хранит их и позволяет строить запросы на PromQL.
Grafana визуализирует метрики как дашборды и корреляции по времени.
Alertmanager доставляет уведомления в мессенджеры и почту, подавляет дубликаты и группирует алерты.
Postgres_exporter публикует метрики PostgreSQL: соединения, транзакции, репликацию, cache hit, background writer, WAL и другое.
connections / max_connections
TPS / deadlocks / temp files
replication lag
cache hit / WAL rate

Визуальный блок

Схема показывает типовой поток метрик: exporters публикуют их, Prometheus собирает, Grafana визуализирует, Alertmanager уведомляет о пороговых или аномальных состояниях.

Практический блок

scrape_configs: - job_name: 'postgres' static_configs: - targets: ['db01:9187'] - job_name: 'node' static_configs: - targets: ['db01:9100']
Рекомендуемый дашборд: • connections / max_connections • tps, commits, rollbacks • cache hit ratio • disk await рядом с pg IO • replication lag • wal generation rate • autovacuum activity
- alert: TooManyConnections expr: pg_stat_activity_count / pg_settings_max_connections > 0.85 for: 5m - alert: HighReplicationLag expr: pg_replication_lag_seconds > 10 for: 2m - alert: AutovacuumStuck expr: pg_vacuum_oldest_xmin_age > 2e9
Сильный дашборд — это не просто много графиков, а визуальные ряды, которые позволяют быстро проверить гипотезу: например, рост latency совпал с checkpoint burst, а не с ростом числа соединений.

Уровень 3

Встроенные средства PostgreSQL

На этом уровне начинается нативная диагностика самой базы: текущая активность сессий, статистика по базам, прогресс операций, I/O и планы выполнения. Это основной рабочий слой DBA и разработчика при разборе медленных запросов и блокировок.

pg_stat_activity pg_stat_database pg_stat_io pg_stat_progress_* EXPLAIN

Теоретический блок

pg_stat_activity показывает серверные процессы, состояние сессии, текущий SQL и wait events.
pg_stat_database даёт агрегированную статистику по транзакциям, чтениям, temp files, deadlocks и конфликтам.
pg_stat_io помогает анализировать чтения и записи по backend type, context и object.
pg_stat_progress_* показывает прогресс VACUUM, CREATE INDEX, COPY, ANALYZE и других долгих операций.
EXPLAIN (ANALYZE, BUFFERS) раскрывает реальный путь выполнения запроса.
state / wait_event
xact_commit / rollback
reads / writes / write_time
estimated rows vs actual rows

Визуальный блок

SELECT pid, usename, state, wait_event_type, wait_event, now() - query_start AS runtime FROM pg_stat_activity WHERE state <> 'idle' ORDER BY runtime DESC; pid | usename | state | wait_event_type | wait_event | runtime ------+---------+--------+-----------------+------------------+---------- 8121 | app | active | IO | DataFileRead | 00:00:14 8127 | app | active | Lock | transactionid | 00:00:09
В идеале actual rows и actual time должны быть предсказуемы, а неожиданный Seq Scan или сильное расхождение estimates с actual — повод копать статистику и индексы.

Практический блок

SELECT state, count(*) FROM pg_stat_activity GROUP BY state; SELECT pid, wait_event_type, wait_event, now() - query_start AS runtime, query FROM pg_stat_activity WHERE state = 'active' ORDER BY runtime DESC LIMIT 20;
SELECT backend_type, object, context, reads, writes, read_time, write_time FROM pg_stat_io ORDER BY write_time DESC NULLS LAST LIMIT 10; SELECT datname, xact_commit, xact_rollback, blks_hit, blks_read, temp_files, deadlocks FROM pg_stat_database;
EXPLAIN (ANALYZE, BUFFERS, VERBOSE) SELECT * FROM orders WHERE customer_id = 42 ORDER BY created_at DESC LIMIT 20; Смотрите на: • rows estimated vs actual • timing по узлам • shared/local/temp blocks • неожиданные scans или rechecks
При long-running запросах полезно сначала смотреть pg_stat_activity, потом планы, а уже затем расширения. Это позволяет быстро понять: проблема в блокировке, в I/O, в плохом плане или во внешнем давлении на сервер.

Уровень 4

Расширения и плагины PostgreSQL

Этот уровень нужен, когда стандартной статистики уже мало. Расширения позволяют собирать агрегаты по запросам, автоматически сохранять планы медленных SQL, анализировать bloat, видимость страниц и состояние buffer cache.

pg_stat_statements auto_explain pgstattuple pg_visibility pg_buffercache

Теоретический блок

pg_stat_statements агрегирует planning/execution statistics по нормализованным SQL и помогает найти тяжёлые запросы не по одному запуску, а по совокупному влиянию.
auto_explain автоматически пишет планы медленных запросов в лог, включая ANALYZE и буферы.
pgstattuple помогает оценить bloat таблиц и индексов, долю dead tuples и реальную плотность хранения.
pg_visibility показывает состояние visibility map и помогает понимать, насколько эффективно vacuum поддерживает видимость страниц.
pg_buffercache позволяет посмотреть, что именно держится в shared buffers.
calls / total_exec_time
total_plan_time
dead_tuple_percent
buffer residency

Визуальный блок

SELECT query, calls, total_exec_time, mean_exec_time, rows, shared_blks_hit, shared_blks_read FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 5; query | calls | total_exec_time | mean_exec_time --------------------------+-------+-----------------+---------------- SELECT ... orders ... | 15420 | 92820 ms | 6.0 ms
Расширения особенно полезны там, где стандартные views показывают симптом, но не дают удобного агрегата по всей рабочей нагрузке или не раскрывают физическое состояние таблицы.

Практический блок

shared_preload_libraries = 'pg_stat_statements,auto_explain' compute_query_id = on CREATE EXTENSION pg_stat_statements; CREATE EXTENSION pgstattuple; CREATE EXTENSION pg_visibility; После изменения shared_preload_libraries нужен рестарт.
SELECT query, calls, total_plan_time, total_exec_time, mean_exec_time, rows, shared_blks_hit, shared_blks_read FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 10; Ищите не только самый медленный запрос, но и запросы с большим суммарным ущербом.
SELECT * FROM pgstattuple('public.orders'); SELECT * FROM pg_visibility_map_summary('public.orders'); Интерпретация: • высокий dead_tuple_percent → autovacuum не успевает • сильный bloat → возможны лишние чтения и плохое использование диска
Глубокая оптимизация почти всегда начинается с pg_stat_statements: он показывает не «самый громкий» SQL, а действительно самый дорогой для всей системы.

Уровень 5

Внутренняя телеметрия: WAL, checkpoints, autovacuum, wait events, репликация

Это самый внутренний и самый мощный слой диагностики. Здесь мы видим, что происходит с записью WAL, работой checkpointer и bgwriter, жизненным циклом мертвых кортежей, ожиданиями backend-процессов и отставанием реплик.

pg_stat_wal pg_stat_checkpointer pg_stat_bgwriter autovacuum replication lag

Теоретический блок

WAL фиксирует изменения до записи в data files и помогает анализировать интенсивность записи и нагрузку на durability.
pg_stat_checkpointer показывает число scheduled/requested checkpoints и затраты времени на запись и синхронизацию.
pg_stat_bgwriter описывает поведение background writer и буферов.
Autovacuum нужен для очистки dead tuples, обновления visibility map и предотвращения transaction ID wraparound.
Replication требует контроля lag, активности sender/receiver и понимания, где именно возникает отставание: send/write/flush/replay.
Wait events показывают, что блокирует прогресс backend: lock, IO, LWLock, client, IPC и другие категории.
wal_bytes / wal_records
num_timed / num_requested
buffers_clean / buffers_alloc
vacuum progress / replay lag

Визуальный блок

SELECT num_timed, num_requested, num_done, restartpoints_timed, restartpoints_req, checkpoint_write_time, checkpoint_sync_time FROM pg_stat_checkpointer; SELECT wal_records, wal_fpi, wal_bytes, wal_buffers_full FROM pg_stat_wal;
Низкоуровневые статистики особенно полезны, когда деградация периодическая: её часто вызывают не отдельные запросы, а циклические фоновые механизмы или нагрузка на WAL и storage.

Практический блок

SELECT num_timed, num_requested, checkpoint_write_time, checkpoint_sync_time FROM pg_stat_checkpointer; SELECT wal_records, wal_fpi, wal_bytes, wal_buffers_full FROM pg_stat_wal; Если num_requested растёт часто, а диск перегружен, проверьте max_wal_size, checkpoint_timeout и профиль bulk writes.
SELECT pid, relid::regclass, phase, heap_blks_total, heap_blks_scanned, heap_blks_vacuumed, index_vacuum_count, num_dead_tuples FROM pg_stat_progress_vacuum; SELECT relname, n_live_tup, n_dead_tup, vacuum_count, autovacuum_count FROM pg_stat_user_tables ORDER BY n_dead_tup DESC LIMIT 10;
SELECT application_name, state, sync_state, pg_wal_lsn_diff(pg_current_wal_lsn(), sent_lsn) AS send_lag_bytes, pg_wal_lsn_diff(sent_lsn, write_lsn) AS write_lag_bytes, pg_wal_lsn_diff(write_lsn, flush_lsn) AS flush_lag_bytes, pg_wal_lsn_diff(flush_lsn, replay_lsn) AS replay_lag_bytes FROM pg_stat_replication; Смотрите, где именно копится отставание.
Autovacuum — критический механизм жизненного цикла таблиц. Если он не успевает, растут dead tuples, bloat, время чтения и риск wraparound. Если отстаёт репликация, нельзя смотреть только на общий lag — нужно разложить его по стадиям send/write/flush/replay.

Диагностические сценарии: как связывать уровни между собой

Этот блок нужен для «самостоятельной работы сверх базы», потому что показывает не просто список инструментов, а типовой маршрут расследования инцидента: от симптома до подтверждённой первопричины.

latency spike locks autovacuum lag replication lag

Кейс 1. Резкий рост latency

1
Уровень 2: на Grafana растёт response time и disk await.
2
Уровень 1: iostat -xz 1 показывает высокий await и %util.
3
Уровень 5: pg_stat_checkpointer показывает рост requested checkpoints и long write/sync time.
4
Вывод: причина не в одном SQL, а в burst write из-за checkpoint pressure.

Кейс 2. Таблица «распухла» и запросы деградируют

1
Уровень 3: растут n_dead_tup и время чтения.
2
Уровень 5: прогресс autovacuum редкий или отстающий.
3
Уровень 4: pgstattuple() подтверждает bloat и долю dead tuples.
4
Вывод: нужно улучшать autovacuum policy, а иногда и пересобирать объекты.

Кейс 3. Подвисшие запросы приложения

1
Уровень 3: pg_stat_activity показывает wait_event_type = Lock.
2
Уровень 3: находим blocker по pg_blocking_pids().
3
Уровень 4: по pg_stat_statements ищем SQL, который чаще всего приводит к блокировкам.
4
Вывод: проблема в конкурентном паттерне транзакций, а не в CPU.

Кейс 4. Реплика отстаёт

1
Уровень 2: алерт по replication lag.
2
Уровень 5: делим lag на send/write/flush/replay.
3
Уровень 1: проверяем сеть или диск standby, если проблема не в отправке.
4
Вывод: отставание может быть вызвано как сетью, так и применением WAL на стороне standby.
